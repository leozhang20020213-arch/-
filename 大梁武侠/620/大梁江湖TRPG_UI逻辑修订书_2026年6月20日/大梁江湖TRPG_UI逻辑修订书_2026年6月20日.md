# 大梁江湖TRPG UI逻辑修订书（2026年6月20日）

用途：投给 Codex 作为下一轮 UI 与逻辑修订依据；不是规则书正文，不替代项目源规则书。

## 一、当前原型结论

当前原型方向正确：本地 Web、专用交锋桌面、事件驱动、规则层输出结果。但截图显示它仍是“单页开发演示桌面”，一进入就是交锋，DM主持台、玩家操作、敌人信息、日志、气骰面板全部压在同一页。下一步必须升级为有入口、有房间、有身份、有玩家端/DM端/背包/敌人详情分区的跑团桌面。

## 二、底层规则逻辑审查

只能基于 README、截图和规则书回填口径审查，不能逐函数确认代码。

### 已符合方向
- 本地 Web 原型定位正确。
- 已有气骰五区：气池、气海、锁气、息库、临气区。
- 已有宣言锁气、截击取消、成招、应招、落果雏形。
- 已有调息、返照、来源失效、DM裁定入口。
- README 明确 UI 动画只读取事件，规则结果来自规则层。

### 不完备
- 缺少开始游戏、创建/加入房间、选择团包、选择身份、选择角色。
- DM/玩家视图切换像调试按钮，不是正式身份权限。
- 玩家、敌人、DM隐藏信息、日志、气骰操作混在一屏。
- 缺少背包/装备/药物/器具界面。
- 缺少团包管理和房间设置页面。
- 交锋阶段要补准备/先后、情景转交锋确认、势变化、轮次结束。
- 敌人详情要分公开卡与DM完整卡。

## 三、必须恢复的既定要求

- 第一阶段是“大梁江湖TRPG专用交锋辅助桌面”，不是通用VTT。
- 入口必须是开始游戏、继续游戏、创建房间、加入房间、团包管理、资料库、设置。
- 玩家端、DM端按身份分流。
- App只辅助交锋、气骰、状态、长效被动、场景轨和日志。
- 势面统一为阴盛、阳盛、合势、圆融、崩溃、失势。
- 六根/内功盘固定为头部/顶门、眼、心、丹田、腰、腿。

## 四、正式页面层级

0. 启动层：首页、开始游戏、继续游戏、加入房间、团包管理、资料库、设置。
1. 房间层：创建房间、加入房间、身份选择、玩家名、旁观设置。
2. 团包层：团包选择、团包详情、样例团包、导入/复制/删除。
3. 角色层：选择角色、创建角色、角色详情、内功/外功/装备/背包。
4. 桌面层：玩家桌面、DM主持台、共享交锋舞台、日志回放。
5. 交锋层：气骰五区、招式、响应、状态、敌人卡、裁定、广播结算。
6. 资料层：规则资料、招式库、状态库、装备库、世界观词条。

## 五、启动与房间设置

启动首页主按钮：开始游戏、继续上次、加入房间。次按钮：团包管理、资料库、设置。不要默认进入样例团包。

创建房间页字段：房间名、主持人名称、团包、游戏模式、是否允许旁观、人数上限。

加入房间页字段：房间码/本地房间列表、玩家名称、身份选择。正式界面玩家不能切到DM。

## 六、玩家端

玩家桌面：顶部显示团包、场景、阶段、轮次、当前势；左侧我的角色卡；中间交锋舞台；右侧敌方公开卡/队友概览；底部气骰五区、招式栏、响应、调息、返照、取物、背包。

玩家不能看到DM隐藏轨、敌人完整技能、隐藏目标、伏兵信息。

玩家交锋流程：选择目标 → 选择招式 → 锁气 → 确认宣言 → 截击窗口 → 成招 → 应招 → 落果 → 反馈。

背包分栏：装备、药物、器具、杂物、临时来源。使用物品必须生成事件。

## 七、DM端

DM房间准备页：玩家席位、角色选择、团包、当前场景、敌人库、公开资料。

DM主持台：左侧场景/玩家/敌人库，中间交锋舞台，右侧追踪器/裁定面板/隐藏信息/广播预览，底部推进阶段和主持按钮。

敌人投入页：公开信息、隐藏信息、入场条件、行为提示、可用招式、状态免疫、掉落/线索。

裁定面板：DM可修改打中/落空/部分成效、伤害、状态、势变化、退场、隐藏信息公开。每次裁定必须生成日志。

## 八、当前截图修改意见

- 顶部DM/玩家视图切换仅保留为开发调试。
- 左侧角色与敌人列表拆成我的角色、队友概览、敌方公开卡。
- 右侧DM主持台移出玩家桌面。
- 中间交锋圆形改成左右阵营交锋舞台。
- 气骰五区绑定真实骰子状态。
- 敌人投入/DM隐藏信息放到DM端独立抽屉或敌人详情页。
- 日志降级为侧栏或抽屉。
- 新增背包/装备/药物入口。

## 九、Codex总提示词

你现在需要修正“大梁江湖TRPG 跑团桌面”原型。不要重做规则，不要重做成通用VTT，也不要只改皮肤。请按照以下“UI逻辑修订书”重构信息架构、页面入口、角色权限、玩家/DM/敌人/背包界面和页面交互。

核心目标：
1. App启动后先进入开始游戏/房间设置/团包选择，而不是直接进入交锋桌面。
2. 玩家端、DM端、敌人信息、背包/装备/药物、资料库必须清楚分层，不允许混在同一个列表里。
3. 当前的DM视图/玩家视图切换只能作为开发调试入口，正式界面要按身份进入。
4. 底层规则继续采用事件驱动：规则层输出事件，UI和动画只读取事件，不参与规则计算。
5. 交锋流程必须符合规则书：场景开始投气骰进气海；同一场景内情景/交锋切换不自动重投；宣言、锁气、截击、成招、应招、落果、势变化必须是阶段状态机。
6. 玩家只能操作自己的角色、自己的气骰、自己的背包和自己可见的信息；DM可以管理场景、敌人、隐藏信息、裁定、广播结算。
7. 敌人信息要分公开/隐藏两层。玩家看到敌人公开卡，DM看到完整敌人详情。
8. 背包界面必须独立：装备、药物、器具、临时气骰来源、来源失效、使用记录都要与交锋流程打通。
9. 每个页面必须有明确入口、返回、主操作、次操作、空状态、错误状态和移动端适配。

请优先修改路由和页面信息架构，再修改组件。不要先做动画。先让页面顺序、身份权限和数据归属正确。
